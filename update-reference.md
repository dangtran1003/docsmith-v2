# Update Reference

Detailed logic for safely updating, re-running, and CRUD'ing on existing docsmith content. Covers re-run protocol, KB inheritance, drift detection, and delete propagation. SKILL.md references this file.

## Why this exists

Docsmith v1.0–1.2 always wrote outputs assuming a fresh slate. That works for the first pass but is dangerous on re-run:

- Re-running `plan` after the team has manually refined it would silently overwrite the work
- Re-running `walkthrough` after product UI changes would mix valid and invalid claims into doc fixes without user oversight
- Deleting a draft from workspace would leave orphan files in the deploy target

v1.3.0 adds three layers of safety:

1. **Re-run protocol** — every output check existence first; prompt 4-option gate
2. **KB inheritance** — update mode reads existing content as knowledge base, proposes deltas
3. **Drift detection** — walkthrough phases verify before fix before capture
4. **Delete propagation** — deploy detects orphan target files

---

## 1. Re-run protocol

### When triggered

Every command that writes a file/folder owned by docsmith checks for existence **before** writing. If the path already has content (file exists with non-zero size, or directory exists with content), trigger the gate.

### The 4-option gate

```
⚠ Already exists: documentation/plan/documentation-plan.md
  Last modified: 2026-01-15 (3 months ago)
  Size: 87 sections, 4521 words
  Originally generated by: docsmith v1.1.0

What do you want to do?

  [1] Update      — keep existing as KB, propose deltas, apply per-item
  [2] Overwrite   — discard existing (archived), generate fresh
  [3] Side-by-side — keep both, new file gets suffix (-v2 or -2026-04-26)
  [4] Cancel      — abort this command

Default action (.docsmithrc.yaml `behavior.on_existing`): prompt
```

The default behavior is configurable. Per the v1.3.0 default: `prompt` (always ask). Users who want speed can set `update` to default to mode 1.

### Mode 1: Update (recommended default for content)

The most important mode — preserves human work. Behavior:

1. Read existing file fully into AI context as **knowledge base**
2. AI generates a **delta proposal**, not a full file:
   ```
   Proposed changes to documentation-plan.md:

   + [NEW] Section 4.3 "Auto-scaling instances" (4 docs proposed)
       Reason: Product released auto-scaling feature 2026-03-10
   ~ [UPDATE] Section 2.1 "Instance types"
       Reason: Add c5.xlarge variant from product changelog
   - [REMOVE?] Section 5.7 "Legacy m1 instances"
       Reason: Marked deprecated in product 2026-02; no docs needed?
       (LOW confidence — please confirm)
   = [KEEP] Section 1, 3, 4.1, 4.2 (untouched)
   ```
3. User approves per-item: `y` (apply), `n` (skip), `e` (edit before apply), `a` (apply all remaining), `q` (stop and save progress)
4. Write merged version
5. Backup original to `archive/<timestamp>/<original-filename>.md`

**Critical rule for AI**: in Update mode, AI must NOT regenerate sections marked `KEEP`. If AI rewrites untouched content (even with "improvements"), it loses the team's manual edits. AI should treat existing content as canonical except where deltas are explicitly proposed.

### Mode 2: Overwrite

Fast but lossy. For when existing content is genuinely outdated and not worth merging.

1. Confirm explicitly: `Type "OVERWRITE" to confirm:`
2. Backup existing → `archive/<timestamp>/<filename>`
3. Generate fresh from scratch
4. Print: "N lines archived → archive/<timestamp>/"

### Mode 3: Side-by-side

For comparison or major rewrite without commitment.

1. New file gets suffix:
   - First side-by-side: `<filename>-v2.md`
   - Second: `<filename>-v3.md`
   - Or with `--suffix-style date`: `<filename>-2026-04-26.md`
2. Both files exist; user manually diffs and merges
3. Note added to both files' frontmatter:
   ```yaml
   ---
   side_by_side_pair: [documentation-plan.md, documentation-plan-v2.md]
   created: 2026-04-26
   ---
   ```
4. After user merges manually, they delete the unwanted file

### Mode 4: Cancel

Abort. No writes. No state changes.

### Configuring default

```yaml
# .docsmithrc.yaml
behavior:
  on_existing: prompt    # prompt | update | overwrite | side-by-side
                         # Default in v1.3.0: prompt (always ask)
                         # Power users may set: update (auto-merge)
```

Even with `on_existing: update`, the gate still shows the delta proposal for user approval per-item — only the initial 4-option prompt is skipped.

## 2. KB inheritance — what AI does in Update mode

When a command runs in Update mode, the AI prompt MUST include this discipline:

### Read first, generate second

1. Read the existing artifact in full
2. Read related artifacts as additional context:
   - `audience` Update → also read existing personas
   - `plan` Update → also read existing audience-profile + sitemap
   - `voice` Update → also read existing drafts to ensure consistency
   - `draft` Update → also read sitemap, voice-chart, ux-text-patterns
3. Identify deltas needed (from new input, e.g. user message or new product info)
4. Generate proposal as 4-kind diff: NEW / UPDATE / REMOVE / KEEP
5. Show proposal, wait for per-item approval

### Per-artifact merge logic

**`audience-profile.md`**
- KEEP all personas unless user says to remove
- UPDATE persona if their context changed (new role, new use case)
- ADD new persona if user mentions one
- REMOVE only with explicit user confirmation
- Always preserve "user research notes" section verbatim

**`documentation-plan.md` + `traceability-matrix.md`**
- KEEP entries for docs that exist in `drafts/`
- UPDATE entries when scope changes (added section, deprecated section)
- ADD new doc entries for new features
- REMOVE only when corresponding draft is also deleted

**`sitemap.md`**
- KEEP existing structure
- ADD new entries in alphabetical/logical position relative to siblings
- UPDATE titles only if normalization rules find inconsistency
- REMOVE entries paired with draft deletion (see § Delete propagation)

**`voice-chart.md` + `ux-text-patterns.md` + `content-scorecard.md`**
- These are usually set once and rarely revisited
- Update mode: AI proposes additions only (new pattern observed in drafts)
- AI does NOT propose changing existing voice rules without explicit user request

**`drafts/<file>.md`** (most sensitive)
- Read entire file as KB
- New input usually targets specific sections (user request: "update the API authentication section")
- AI proposes change to that section only
- Other sections: KEEP verbatim (do NOT "improve" them)
- Preserve all `<!-- VIDEO ... -->` markers and image refs unless explicitly told to remove

**`walkthrough/test-cases/`**
- Each test case is keyed by ID (e.g., `TC-INST-CREATE-002`)
- KEEP test cases that still match current draft content
- UPDATE assertions when draft step text changes
- ADD test cases when new draft steps are added
- REMOVE test cases when corresponding draft step is deleted

### What AI must NOT do in Update mode

- Regenerate untouched content "for consistency"
- Reorder sections without user request
- Apply newer voice rules retroactively to old content (unless user opts in)
- Silent merge of similar sections — if 2 sections look like duplicates, FLAG it, don't auto-merge

## 3. Drift detection — walkthrough 3-phase

The default `walkthrough` command was: verify + auto-fix + capture, all in one. v1.3.0 splits this into phases with a user gate between them.

### Phase A — VERIFY (read-only)

```bash
/docsmith walkthrough MyProduct --check
```

AI:
1. Read all drafts in scope
2. Build test cases from drafts (or load from `walkthrough/test-cases/`)
3. Open browser, navigate per test cases
4. For each assertion: check if doc claim matches UI
5. **Do NOT modify drafts. Do NOT capture screenshots.**
6. Output `documentation/walkthrough/drift/<timestamp>/drift-report.md`
7. Exit

**Drift report format**: see [templates/DRIFT_REPORT_TEMPLATE.md](templates/DRIFT_REPORT_TEMPLATE.md).

Each drift item has:
- `id` — stable identifier
- `doc` + `section` — where in draft
- `assertion` — what doc claims
- `observed` — what UI shows
- `category` — `label-change` / `field-added` / `field-removed` / `flow-change` / `missing-element` / `unclear`
- `confidence` — `HIGH` / `MEDIUM` / `LOW`
- `suggested_fix` — proposed doc edit (HIGH/MEDIUM only; LOW gets `null`)

### Gate — User decides per item

User reviews `drift-report.md`, fills `decisions.yaml`:

```yaml
# documentation/walkthrough/drift/2026-04-26-103044/decisions.yaml
items:
  - id: drift-001
    decision: auto-fix         # apply suggested_fix as-is
  - id: drift-002
    decision: manual-fix       # user wrote a custom fix in decisions.yaml
    custom_fix: |
      Replace step 5 entirely with:
      "5. In the form, set the network mode (defaults to 'shared'):
          - Shared: ...
          - Dedicated: ..."
  - id: drift-003
    decision: product-bug      # doc is correct, UI has regression
    notes: "Submit button missing — opened JIRA-1234"
  - id: drift-004
    decision: skip             # neither doc nor product wrong, just ignore
    notes: "Cosmetic — button moved 4px, irrelevant"
```

`decisions.yaml` is human-edited. Keep open in editor while reviewing the report. AI provides defaults: `auto-fix` for HIGH confidence, `skip` for LOW. User overrides as needed.

**Optional shortcut**: `--auto-apply-high-confidence` flag skips the gate and auto-applies HIGH-confidence fixes directly. Use only when speed matters more than safety (e.g., daily CI runs) — and even then, capture phase still gets a chance to fail loudly if something looks wrong.

### Phase B — UPDATE DOC (only if gate passed)

```bash
/docsmith walkthrough MyProduct --apply
```

AI:
1. Load latest drift report + decisions.yaml
2. For each item with `decision: auto-fix`: apply `suggested_fix` to draft
3. For each item with `decision: manual-fix`: apply `custom_fix` to draft
4. For each item with `decision: product-bug` or `skip`: do NOT touch draft
5. Show diff to user before commit
6. On confirm: write changes; save patches in `drift/<ts>/auto-fixes-applied.diff`

### Phase C — CAPTURE

After phase B, AI continues into normal walkthrough capture:
1. Re-build test cases (now that drafts are updated)
2. Capture screenshots per existing capture plan
3. Replace `placehold.co` placeholders
4. Standard walkthrough output

### Default behavior (no flags)

```bash
/docsmith walkthrough MyProduct
```

Runs A → gate → B → C in sequence with confirmation prompts. Backward compatible with v1.2.x usage.

### Skip flag

```bash
/docsmith walkthrough MyProduct --skip-drift
```

Run as v1.2.x did: capture only, no drift detection. For first-time runs where there's no existing draft to drift from.

### Re-running drift after marking product-bugs

Subsequent walkthrough runs read previous `drift/<ts>/decisions.yaml`. Items previously marked `product-bug` are tracked in `drift/active-product-bugs.yaml` (rolled up from latest decisions). New runs:

- Items with same drift signature as a tracked product-bug → don't re-flag (skip in report)
- After product-bug fixed (UI now matches doc again) → automatically remove from `active-product-bugs.yaml`
- Output: `Tracked product bugs: 3 active. 1 resolved this run (drift-003).`

This keeps drift reports clean over time without losing visibility into ongoing product issues.

## 4. Delete propagation

### The problem

User deletes `documentation/drafts/en/instances/legacy.md` because the feature was removed from product. After `deploy`, `<target>/docs/instances/legacy.md` still exists — orphan.

### Detection

`deploy --dry-run` adds a section to plan output:

```
Orphan files in target (no source in workspace):
  ? docs/instances/legacy.md
  ? docs/api/v1-deprecated.md
  ? static/img/mycloud/instances/legacy-form.png

To delete: re-run with --sync-deletes
To keep:   move out of target docs path, or add to docusaurus exclude
```

### Sync-deletes flag

```bash
/docsmith deploy MyProduct --sync-deletes --dry-run
```

Plan now includes `delete` actions. User reviews, then:

```bash
/docsmith deploy MyProduct --sync-deletes
```

Executes deletes after copies. Deletes are recorded in audit trail:

```yaml
# deployments/<ts>/manifest.yaml
deletes:
  - target: docs/instances/legacy.md
    reason: "no source in workspace"
    backup: deployments/<ts>/deleted/instances/legacy.md
```

Deleted files are backed up inside the deploy audit folder for 30 days (or until next deploy with `--sync-deletes` purges old audits — configurable).

### Default: never delete

Without `--sync-deletes`, deploy never deletes from target. Orphans are reported only. This is the safe default.

### Sitemap consistency

When a draft is deleted:
1. AI suggests removing the corresponding sitemap entry (next time sitemap is run or in deploy plan)
2. If sitemap still references a deleted draft: `verify` flags it as broken-link
3. `categorize` skips category generation for deleted folder

## 5. CRUD optimization summary

This release makes thêm/sửa/xoá deterministic:

### Add (thêm)

- **New doc**: edit `sitemap.md`, run `/docsmith draft <product>` — gate triggers for sitemap (Update mode), AI generates only the new doc
- **New section in existing doc**: edit draft directly, run `/docsmith verify` to check; or use `/docsmith draft <product> --section <name>` for AI-generated section
- **New screenshot**: insert `placehold.co` in draft, run `/docsmith walkthrough --skip-drift` (only captures missing placeholders) or rely on default `walkthrough` (which also re-verifies — slower but safer)
- **New video**: insert `<!-- VIDEO ... -->` marker, run `/docsmith record`

### Update (sửa)

- **Content edit**: edit draft, run `/docsmith verify <doc>` to check
- **AI-assisted edit**: `/docsmith draft <product>` → Update mode → propose delta
- **After product UI change**: `/docsmith walkthrough --check` → review drift → `/docsmith walkthrough --apply`
- **After upstream voice change**: re-run `/docsmith voice` in Update mode → flag inconsistencies in existing drafts; user decides per-doc whether to apply

### Delete (xoá)

- **Doc**: delete file from `drafts/`, edit sitemap, run `/docsmith deploy --sync-deletes --dry-run` to preview, then `--sync-deletes` to apply
- **Section**: edit draft directly, re-run `/docsmith walkthrough --check` to find orphan test cases / images
- **Screenshot**: delete `.png` and remove markdown ref; or delete from draft and let `verify` flag orphan image file
- **Video**: same as screenshot, plus delete `.mp4`

### Performance hints

- `walkthrough --check` (Phase A only) is fast — under 30s for moderate doc set; use this for daily drift monitoring
- `walkthrough --skip-drift` (capture only) is fast on incremental adds — only chases new placeholders
- `verify <glob>` scopes verification to changed files; much faster than full project verify
- `deploy --dry-run` is always cheap; never skip it before apply

## 6. Audit trail

Every operation that modifies user content is recorded:

```
documentation/
├── archive/<timestamp>/                # Re-run protocol backups (Mode 1, 2)
│   └── <original-filename>
├── walkthrough/drift/<timestamp>/      # Drift detection runs
│   ├── drift-report.md
│   ├── decisions.yaml
│   ├── auto-fixes-applied.diff
│   └── product-bugs.md (rolled into active-product-bugs.yaml)
├── walkthrough/active-product-bugs.yaml  # Cross-run tracker
└── walkthrough/executions/<timestamp>/  # Walkthrough runs (existing)

deployments/<timestamp>/
├── manifest.yaml                       # Includes deletes if --sync-deletes
├── deleted/                            # Backup of removed target files
└── (existing files: target-config.yaml, diff.md, pre-deploy-state.txt)
```

Retention: archive folders kept until manually cleaned. Drift folders kept indefinitely (small files). Deployment audit folders retained per `deploy.audit_retention_days` config (default: forever).

## 7. Config additions in v1.3.0

```yaml
# .docsmithrc.yaml additions

behavior:
  on_existing: prompt              # prompt | update | overwrite | side-by-side
  drift_default_action: prompt     # prompt | auto-apply-high-confidence
  side_by_side_suffix_style: numbered   # numbered (-v2) | date (-2026-04-26)

deploy:
  sync_deletes: false              # default false; opt-in only
  audit_retention_days: 0          # 0 = forever
```

All optional. Defaults preserve v1.2.x behavior except where new safety nets are pure additions.
